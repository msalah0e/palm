Section 01

Executive Summary

The strategic case for tamr as the first AI tool package manager and matrix hub.

tamr is the first CLI that combines AI tool package management, API key vault, unified configuration, and cost tracking into a single, zero-dependency binary. After exhaustive research covering 130+ AI CLI tools across 13 categories, no competing AI tool manager was found. The closest adjacent tools (Ruler, LNAI, coder-config) address fragments of the problem but none attempt the full package-management layer.


The v0.3.0 release stripped the Homebrew wrapper and promoted AI commands to top-level, focusing tamr squarely on its unique value proposition. This report covers the research and architecture for v0.4.0+ — the "Matrix Hub" vision — transforming tamr from a tool installer into a comprehensive control plane for the AI development toolkit.

Key Architecture Patterns from Research

mise

Hybrid embedded + external registry. Config hierarchy (global → workspace → project). Lock files for reproducibility. Adopted — config hierarchy pattern.

proto

WASM plugins for extensibility. Not adopted — TOML schema plugins cover 97% of tamr use cases without WASM complexity.

volta

Shim-based version management. Not needed — tamr does not manage tool versions, only installs and runs.

homebrew

Tap system for community registries. Bottle system for pre-built binaries. Adopted — plugin registry pattern.

Section 05

Gap Analysis

Eight critical gaps in the AI tool ecosystem that tamr is uniquely positioned to fill.

HIGH Key Sprawl

API keys scattered across .bashrc, .env, .env.local, macOS Keychain, 1Password. No single source of truth. Developers waste time managing the same keys in 5+ places.

CRITICAL Tool Fragmentation

Every tool has its own config format: CLAUDE.md, .aider.conf.yml, .cursorrules, codex.md, GEMINI.md. No "write once, run everywhere" for project context.

HIGH Cost Visibility

Each AI provider has its own billing dashboard (Anthropic Console, OpenAI Platform, Google Cloud). No unified real-time cost view across all tools and providers.

MEDIUM Model Confusion

Each tool defaults to different models (Claude Sonnet, GPT-4o, Gemini Flash). No central way to declare "use claude-opus-4-6 everywhere" for a project.

MEDIUM No Unified History

Claude Code sessions in ~/.claude/, Aider logs in git history, Cursor in its SQLite DB. No cross-tool session visibility or searchable history.

HIGH No Project-Level Config

No way to codify "this project uses Claude Code + Aider + Codex with these specific settings, models, and key bindings." Every new team member starts from scratch.

HIGH Enterprise Deployment

No proxy support, no budget limits, no centralized governance for AI CLI tools. Enterprise teams cannot enforce spending policies or audit tool usage.

MEDIUM No Benchmarking

No standardized way to compare coding agents on your own codebase. Which tool completes a refactoring task faster, cheaper, and more accurately? Nobody knows.

Section 06

Matrix Hub Features

Eight features that transform tamr from a tool installer into a comprehensive AI development control plane.

tamr workspace

v0.4 LOW EFFORT

Project-level .tamr/workspace.toml defining required tools, preferred models, key references, and team settings. Commit it to git. New team members run tamr install and get the full AI toolkit configured.

tamr context

v0.4 MED EFFORT

Single .tamr/context.md transpiled to CLAUDE.md, .aider.conf.yml, .cursorrules, codex.md, and more. Write your project context once — tamr generates the tool-specific config files. "Write once, configure everywhere."

tamr history

v0.5 LOW EFFORT

Unified session log capturing tool name, project, duration, token usage, and estimated cost. Searchable and filterable. Finally answer: "How much did I spend on AI coding this week?"

Shell Integration

v0.5 LOW EFFORT

eval $(tamr env) exports all API keys as environment variables. No more tamr run wrapper required. Add it to .zshrc and every terminal session has keys available.

tamr proxy

v0.6 HARD

Local HTTP proxy intercepting all LLM API calls. Logs every request/response, calculates token costs in real-time, enables request routing and caching. THE foundational feature for cost visibility and enterprise governance.

tamr matrix

v0.6 MED EFFORT

Terminal dashboard built with bubbletea TUI. Shows installed tools with status, active sessions, real-time cost counters, model usage, and key health. Like htop for your AI development toolkit.

tamr budget

v0.7 HARD

Spending limits per provider, per tool, per project. Configurable alerts at thresholds (80%, 90%, 100%). Hard enforcement via the proxy layer. Essential for teams and enterprises managing AI spend.

tamr benchmark

v1.0 HARD

Run multiple coding agents on the same task. Compare completion time, token usage, cost, and output quality. Generates a report showing which tool performs best for your specific codebase and use case.

Section 07

Offline Mode Design

Full functionality without network access. Air-gapped deployments, flight mode, and local-first operation.

  • ◆
    --offline global persistent flag — Set on the root command. Persists across sessions. All registry, search, and info commands work offline via embedded TOML (go:embed).
  • ◆
    tamr fetch <tool> — Pre-download packages to local cache at ~/.cache/tamr/. Supports all backends (brew bottles, pip wheels, npm tarballs, Go binaries). Idempotent.
  • ◆
    tamr bundle export/import — Create portable archives for air-gapped transfer. Bundles include tools, model weights, and encrypted key references. USB-transferable.
  • ◆
    Network auto-detection — Graceful degradation when offline is detected. Falls back to cached registry and local packages. Clear indicators in CLI output when operating in degraded mode.
  • ◆
    tamr models fetch/export/import — Manage LLM model files for local runners. Fetch GGUF/MLX models while online. Export to bundle for offline use. Integrates with Ollama and llama.cpp.
  • ◆
    Embedded registry via go:embed — All 130+ tool definitions compiled into the binary. tamr search, tamr info, and tamr list work with zero network access, always.
Section 08

Parallel Execution

Concurrent tool installation and operations with progress visualization. The "badass mode."

Implementation

  • ◆
    golang.org/x/sync/errgroup with SetLimit for bounded concurrency
  • ◆
    github.com/vbauerster/mpb/v8 (Multi Progress Bar) for parallel visualization
  • ◆
    Backend-aware parallelism: serialize within same backend (brew), parallelize across different backends
  • ◆
    Default concurrency: 4, configurable via --concurrency N or config.toml

Applies To

  • ◆
    tamr install a b c — Install multiple tools concurrently
  • ◆
    tamr update --all — Update all installed tools in parallel
  • ◆
    tamr doctor — Check all tool health concurrently
  • ◆
    tamr fetch --all — Pre-download all packages for offline use
